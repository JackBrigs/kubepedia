# Отчёт по изменениям — `/learn` (2026-07-20)

Петля самоулучшения: детерминированный сборщик сигналов + сверка с тегнутым исходником → одно
сфокусированное изменение по контракту. Ниже — заходы дня, что изменено и приоритезированный бэклог.

## Заход 3 (этот) — простой при добавлении/замене/удалении нод

**Раздел «Service impact» в процедуру добавления/замены/удаления нод (`nodes-add-replace.md`).**
- **Почему:** #1 остатка бэклога P0-A.
- **Сверено с источником** (`roles/remove_node/pre_remove/tasks/main.yml`, v2.29.0 — добавлен `type: code`):
  remove гоняет `kubectl drain --force --ignore-daemonsets --grace-period 300 --timeout 360s
  --delete-emptydir-data`, затем `reset`; add (`scale.yml`) существующие ноды **не дренит**.
- **Добавлено:** add-worker — низкий impact (новая нода джойнится, нагрузки не трогаются); remove —
  дизраптивно на целевой ноде (поды эвиктятся/реджойнятся, `--force` сносит standalone-поды, emptyDir на
  ноде уничтожается, DaemonSet остаются); `allow_ungraceful_removal=true` пропускает ожидание drain;
  CP/etcd — по одной ноде, держать кворум. Не откат.
- **Итог:** валидатор PASS, версии PASS; сигнал 11 → 10.

## Заход 2 — простой при последовательном апгрейде Kubespray

**Раздел «Service impact» в центральную прод-процедуру — последовательный апгрейд Kubespray
(`upgrade-sequential.md`).**
- **Почему:** топ-1 бэклога P0-A. Самая массовая мутирующая операция (апгрейд всего кластара по минорам),
  а явного раздела о простое не было.
- **Сверено с источником** (`playbooks/upgrade_cluster.yml`, v2.29.0 — добавлен как `type: code`-источник):
  порядок плеев — control-plane идёт `serial: 1` (по одной ноде), воркеры/calico батчами `serial` (деф. 20%),
  graceful = cordon → drain → uncordon в pre/post-upgrade.
- **Добавлено (по фактам плейбука):** CP рестартится по одной ноде (при ≥2 CP API жив; одиночный CP теряет
  API); etcd — rolling с сохранением кворума; воркеры дренятся батчами 20% → одиночные реплики и поды без PDB
  ловят downtime, а жёсткий PDB может **застопорить drain**; при `system_upgrade_reboot: on-upgrade` нода
  ребутится на обновлении пакетов; unsafe-путь (`upgrade_cluster_setup=true`) пропускает drain и мигрирует
  static-pods сразу — жёстче для нагрузок. Не откат: состояние сохраняется; каждый минор — отдельное окно.
- **Итог:** валидатор PASS (0 hard / 0 warnings), версии PASS; сигнал закрылся (12 → 11).

## Заход 1 — восстановление control-plane

**Раздел «Service impact» в процедуру восстановления control-plane (`recover-control-plane.yml`).**
- Сборщик пометил её как мутирующую без явного раздела о простое — прямой сосед восстановления etcd.
- Сверено с `playbooks/recover_control_plane.yml` (v2.29.0): пересборка членства etcd + переджойн CP на
  выживших нодах, **без отката состояния**; running-нагрузки живут. Это incident recovery, не рутинный change.
- Валидатор PASS; сигнал закрылся (13 → 12).

## Бэклог на следующие заходы (ранжировано)

## Метод (зафиксирован в скилле `/learn`)
Сигнал → **сверка с тегнутым исходником** → одно изменение → пайплайн валидации (PASS) → русский коммит →
push. За заход — одно улучшение (token-бюджет под контролем). KDS-база не переделывается; ITIL/отчёты — слои
вывода.
