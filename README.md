# Kubepedia — версионно-осведомлённая база знаний по Kubespray

**Kubepedia** — это AI-first граф знаний о Kubernetes, Kubespray и компонентах,
которыми управляет Kubespray. Каждый факт привязан к конкретным версиям Kubespray,
к которым он применим, и подтверждён кодом соответствующего git-тега.

- Строится строго по **коду тегов Kubespray** (не блоги и не `master`)
- **Версионно-осведомлён**: один документ на сущность, значения — таблицей по релизам
- **Атомарный и графовый**: маленькие документы, связанные типизированными связями
- **Машиночитаемый индекс**: отвечает на базовые вопросы без LLM
- **Проверяется**: каждый документ проходит `scripts/validate_kds.py` перед вливанием

## Быстрый старт

Смотрите граф знаний в [`kb/`](kb/) — каждый документ самодостаточен.

Запросы к сгенерированному индексу (JSON Lines) без всякого LLM:

```ShellSession
# все управляемые компоненты и их диапазон версий
jq -r 'select(.type=="component") | "\(.id)\t\(.component_version)"' index/documents.jsonl

# всё, что покрывает Kubespray v2.31.0
jq -r 'select(.kubespray_version | test("2.31")) | "\(.type)\t\(.id)"' index/documents.jsonl

# какой run-тег перегенерирует сертификаты etcd?
grep -rl "etcd certificates" kb/kubespray/ansible-tags/
```

Перегенерация индекса и валидация базы:

```ShellSession
pip install pyyaml jsonschema
python3 scripts/generate_index.py
python3 scripts/validate_kds.py
```

## Использование с LLM

Документы Kubepedia самодостаточны (frontmatter + тело + источники), поэтому база
хорошо ложится в retrieval. Общий рецепт для любого ассистента: дайте ему
содержимое `kb/` (при желании — `index/`), затем задавайте **версионные** вопросы
(«какой etcd в Kubespray v2.31.0 для Kubernetes 1.35?»).

**Рекомендуемый системный промпт** (задаёт «reference-режим»):

```text
Ты отвечаешь ТОЛЬКО по документам Kubepedia, которые тебе даны.
Всегда указывай версию Kubespray, к которой относится ответ, и ссылайся на
документ-источник (id/путь). Если факта в предоставленных документах нет —
скажи об этом прямо и не выдумывай. Версии/значения бери из таблиц внутри
документов, а не «по памяти».
```

Пять популярных инструментов:

1. **Claude** (claude.ai → Projects, либо Claude Code в терминале). Создайте
   Project и добавьте каталог `kb/` в его knowledge; или запустите `claude` прямо
   в клоне репозитория — модель читает файлы локально. Вставьте системный промпт
   выше в инструкции проекта.
2. **ChatGPT** (OpenAI). Соберите `kb/` в архив и загрузите в Custom GPT (или в
   Project) с включённым file search; в инструкции GPT вставьте системный промпт.
3. **Google Gemini** (gemini.google.com / AI Studio). Большое контекстное окно
   позволяет приложить много файлов `kb/` напрямую и сразу спрашивать; для
   объёмных выборок используйте AI Studio.
4. **NotebookLM** (Google). Добавьте документы `kb/` как sources — отвечает
   строго по источникам и даёт цитаты, что идеально для проверяемости фактов
   Kubepedia.
5. **Microsoft Copilot**. Приложите нужные документы `kb/` (или ссылку на файл в
   репозитории) к запросу и спрашивайте с тем же системным промптом.

Для программного доступа (RAG / векторный / graph-поиск) отдавайте модели
markdown из `kb/` как чанки и `index/documents.jsonl` — как метаданные для
фильтрации по версии и типу.

## Требования

- **Python 3** с **PyYAML** (и **jsonschema** для полной проверки метаданных)
- `jq` (опционально) для запросов к индексу
- Локальный чекаут тега Kubespray при расширении базы — факты сверяются по коду
  источника, а не додумываются
